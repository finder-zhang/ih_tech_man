智能家居系统

路由器内部主程序与控制程序的通讯协议

一  概述

智能路由器的角色，是整个家庭的控制中心，手机客户端无论在外网还是在内网，控制信息都要传给路由器的主程序来处理。由于受控器件的多样性（有串口控制与WIFI控制两种）与种类易变性（某个器件使用串口控制，但很可能会改成WIFI控制），

所以有一条中间的控制程序，该程序把控制各种器件的动作封装起来，向路由器主程序提供统一的控制接口，主程序只需要使用统一的控制命令与控制程序交互，即可实现器件的控制。

二  通讯数据形式

本协议采用JSON字符串形式，每一个控制数据包为一行JSON字符串表示的JSON对象，并以换行符(“\n”)结尾，这个结尾必须得有。

 

JSON的格式如下：

{

         “a”: “xxx”,

    …

}

其中字段”a”表示 控制/返回 命令，根据不同的命令，附加不同的参数值。

 

名称协定：

主程序： 路由器或平板主机上的服务器程序

控制程序：与主程序通信并控制物理设备的程序

zigbee程序： 运行在zigbee模块上的程序

模块唯一ID请求 a=controllerIdRequest

主程序请求获取zigbee模块的唯一标识符。主程序将使用该唯一标识符作为主程序自身的唯一标识符。

数据： 无

示例数据包：{“a”: “controllerIdRequest “}

 

模块唯一ID返回 a=controllerIdResponse

串口程序发送给主程序zigbee的设备唯一标识

数据：

字段名

说明

备注

controllerId

串口返回的zigbee模块唯一识别符

 

 

示例数据包：{“a”: “controllerIdResponse “,”controllerId”:”abcdef-12345678-09900000”}

 

开启/关闭入网状态请求 a=openJoinModeRequest

需要注册设备时，主机程序发送该协议

数据：

字段名

说明

备注

time

开启时间

0-255

以秒为单位，其中0表示关闭入网状态，255表示常开入网状态。

type

允许入网的设备类型

 

示例数据包：{“a”:” openJoinModeRequest” , ”time”:255 , ”type”:1 }

 

设备入网返回 a=deviceJoinResponse

当有设备请求入网时，串口程序向主程序发送该协议。

 

数据：

字段名

说明

备注

devicePid

设备的物理id

 

type

设备类型

与服务接口里的type一致

 

示例数据包：{“a”: “deviceJoinResponse “,” devicePid”:” abcdef-12345678-09900000”,”type”:1}

 

 

改变设备状态请求 a=setDeviceStateRequest

主机请求串口程序改变某个设备的状态

数据：

字段名

说明

备注

devicePid

物理设备ID

 

type

设备类型

Int 型

state

设置的状态

 

备注：state的类型为Object 型，根据不同的type 有不同的数据类型，关于各种type 的state的类型，

详见《服务器通讯协议》的设备状态部份

示例数据包：{“a”: “setDeviceStateRequest “,” devicePid”:” abcdef-12345678-09900000” , ”state”:{xxx,xxx,xxx}  }

 

 

改变设备状态返回 a=setDeviceStateResponse

串口程序响应主程序状态变更请求的协议

数据：

字段名

说明

备注

result

操作结果

1 操作成功

0 操作失败

devicePid

物理设备ID

 

state

最终状态

 

msg

错误提示

当操作失败时，可以设置一个错误提示

 

示例数据包：{“a”: “setDeviceStateResponse “,”result”:0,” devicePid”:”xxxxx”,”state”:”1”,”msg”:”物理设备不存在”}

当物理设备断开连接时，由串口程序通知主程序。注意，只要有设备断开时就应该发送该协议给主程序，而不是只有在主程序请求时才返回。

说明

devicePid
